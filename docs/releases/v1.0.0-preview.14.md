Base release: `v1.0.0-preview.13`

Release target: the `main` commit that includes this release-prep document. Do

Asset: `qortium-preview.zip`

## Purpose

This prerelease packages the chat-history fix that keeps a group's messages
visible after a member leaves, and the minter synchronization-wedge work that
stops a node that is merely behind from minting a competing fork, keeps a node
retrying synchronization on its own, and automatically recovers a node that is
already stuck on a short self-minted fork. It also adds two group read endpoints
that complete the symmetric kick/ban read surface for Q-Apps.

## Release Highlights

- Group chat messages from members who have left a group are no longer hidden.
  The node previously filtered group chat down to only senders who are members
  right now, so anything said by someone who later left vanished instantly; now
  group messages stay visible to everyone and simply expire on the normal
  retention schedule. This affects the live `/chat/messages` and message-count
  endpoints and the chat WebSocket.
- A minting node that has fallen behind no longer mints its own competing block
  while a healthy, up-to-date peer is visibly ahead of it — it waits and
  synchronizes instead — closing the path that let a behind minter manufacture a
  short side-chain. A peer that is genuinely ahead is now recognized by height
  alone, removing a timing quirk that previously let this safeguard miss.
- The synchronizer now retries on its own on a periodic heartbeat instead of
  only reacting to peer chain-tip broadcasts, so a node can no longer quietly
  stall when those messages stop, and a single late/failed message no longer
  drops the rest of a batch of peer updates.
- A bounded fork-recovery watchdog automatically rescues a node that is already
  stuck on a short self-minted fork: when its tip is old, several distinct
  healthy peers are clearly ahead, and it has been unable to catch up for
  several minutes, it discards its own top block (strictly depth-capped) and
  requests synchronization so the normal, fully-validated download rejoins the
  real chain. It only ever discards its own blocks, never accepts unverified
  ones, stays inactive on a healthy or legitimately-isolated node, and can be
  disabled in settings (`recoveryWatchdogEnabled`).
- Two new group read endpoints complete the symmetric kick/ban read surface for
  Q-Apps: `GET /groups/kicks/{groupId}` returns all confirmed kicks in a group
  (with optional member-address, timestamp-range, pagination and sort filters),
  and `GET /groups/bans/member?address=` returns all current bans for an address
  across every group. Both validate their inputs and return the same data shapes
  as the existing companion endpoints.

Notes:
- This is a public preview prerelease. Preview accounts, data, and balances are
  not permanent.
- The synchronization-wedge changes are all in the safe direction: a node only
  ever defers minting, retries synchronization, or discards its own short fork;
  no block validation, weighting, or timestamp rules change, and re-adoption of
  the network chain always goes through the normal fully-validated sync path.
- The fork-recovery watchdog defaults to enabled for previewnet; operators can
  set `recoveryWatchdogEnabled` to false until it is vetted for mainnet use.
